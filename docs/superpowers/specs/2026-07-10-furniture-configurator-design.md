# Furniture Configurator & Catalog — Design

## Контекст

Домашнє завдання курсу fwdays Academy "Agentic Engineering: Greenfield" вимагає невеликого власного проєкту, проведеного через повний інженерний цикл (специфікація → план → реалізація → верифікація), із застосуванням agentic-практик — а не масштабного продукту.

Ідея проєкту виникла з реальної проблеми: завод з виробництва меблів використовує 1С з трьома неузгодженими базами, довільний (нестандартизований) ввід атрибутів (напр. "кашемір" vs "кишемір" трактуються як різні продукти), і повністю ручний процес узгодження замовлень між філіалами й заводом.

Повна проблема — це кілька незалежних підсистем: конфігуратор/BOM, синхронізація трьох 1С-баз, трекінг фізичного стану виробу (виробництво/склад/дорога/філіал/продано), рольовий доступ для відділів, аналітика для топ-менеджменту, автоматизація замовлень філіал→завод. Для домашнього завдання обрано **один зріз**: конфігуратор виробів + стандартизований каталог компонентів, з валідною специфікацією (BOM) як результатом.

## Скоуп

**Що робимо:**
Веб-застосунок на Next.js, де менеджер конфігурує виріб (тумбочка-тип: верх + низ + фасад + корпус + фурнітура), система валідує розміри (стандартні/індивідуальні), перевіряє, чи така комбінація вже є в каталозі стандартних виробів, і видає валідну специфікацію (BOM) — список компонентів з SKU, готовий передати у виробництво. Адмін керує довідниками (кольори, матеріали, стандартні розміри, фурнітура) через контрольовані списки (dropdown), що усуває проблему дублікатів на кшталт "кашемір"/"кишемір" — вибір завжди йде зі списку значень у БД, а не вільним текстом.

**Свідомо поза скоупом цього демо** (майбутні кроки, не реалізується):
- Синхронізація трьох баз 1С.
- Трекінг фізичного стану виробу (виробництво → склад → дорога на філіал → філіал → продано).
- Автоматизація процесу замовлення філіал→завод (заміна ручного email-флоу).
- Аналітика/KPI для топ-менеджменту.
- Кухні та інші багатокомпонентні вироби (шафи, шухляди тощо) — конфігуратор підтримує лише один тип виробу (тумбочка-тип: верх/низ/фасад/корпус/фурнітура).
- Перевірка фізичної наявності на складі — це демо про конфігурацію/BOM, не про інвентар.
- Повноцінна автентифікація — вхід за роллю реалізується як проста hardcoded логін-форма з двома захардкодженими користувачами (manager/admin), без реальної системи авторизації.

## Технічний стек

Next.js (App Router) + TypeScript, повний стек в одному застосунку. Prisma ORM + SQLite для персистентності. Доменна логіка (валідація, генерація BOM, пошук збігів у каталозі) реалізована як чисті TypeScript-функції без залежності від Next.js чи Prisma — ізольовані та юніт-тестовані.

Обґрунтування вибору (серед розглянутих варіантів — Next.js+SQLite з чистою доменною логікою; Next.js UI + окремий сервіс правил; config-driven JSON-правила): для одного типу виробу окремий сервіс правил або JSON-driven конфігурація — зайва інфраструктурна складність (порушує YAGNI). Один застосунок з чітко ізольованою доменною логікою дає повний інженерний цикл за розумний час і залишає домен тестованим незалежно від UI/БД.

## Модель даних (Prisma)

**Довідники (контрольовані списки, керує адмін):**
- `Color` — id, name (унікальне, напр. "Кашемір")
- `Material` — id, name (напр. "ЛДСП", "МДФ")
- `HardwareItem` — id, name, sku (напр. "Ручка хром", "Напрямна push-to-open")
- `StandardSize` — id, dimensionType (width/height/depth), valueCm (напр. 40, 60)

**Виріб і компоненти:**
- `ProductType` — напр. "Тумбочка" (виокремлено як сутність на випадок розширення в майбутньому, хоча в цьому демо підтримується лише один тип)
- `PartSpec` — правило для ролі деталі в межах `ProductType`: partRole (top/bottom/facade/corpus), allowsCustomSize (bool), minCm/maxCm (діапазон для індивідуального розміру)
- `CatalogEntry` — збережена **стандартна** комбінація (партія): productType, конкретні id компонентів (facadeColorId, facadeMaterialId, sizeWidth, sizeHeight тощо), sku, isStandard (bool)
- `HardwareOption` — зв'язка `CatalogEntry` ↔ `HardwareItem` (обрана фурнітура для конкретного BOM)

Розмір "верх"/"низ" — або посилання на `StandardSize`, або довільне число, валідоване в доменній логіці (не в БД) в межах `PartSpec.minCm..maxCm`.

**Результат роботи конфігуратора** (для MVP не обов'язково персистувати, але корисно для демо):
`ConfigurationRequest` — обраний користувачем набір компонентів і розмірів → результат валідації (ok/errors) → згенерований BOM (список частин з SKU) → посилання на `CatalogEntry` при збігу, або позначка "нова індивідуальна комбінація".

## Доменна логіка (`lib/configurator/*`)

Чисті функції без залежності від Next.js/Prisma — приймають прості дані, повертають результат; головний об'єкт юніт-тестування.

**`validateSize(partSpec, requestedSize): ValidationResult`**
- Значення збігається зі стандартним зі списку → ОК, `isStandard: true`.
- Значення не стандартне, але `allowsCustomSize` і в межах `[minCm, maxCm]` → ОК, `isStandard: false`.
- Значення поза межами діапазону → помилка з конкретним повідомленням (напр. "розмір 25см поза допустимим діапазоном 30-80см").
- `allowsCustomSize = false` і значення не стандартне → помилка.

**`buildBOM(selection): BOMResult`**
- Валідує кожен розмір через `validateSize`.
- Є помилки → `{ valid: false, errors: [...] }`, BOM не формується.
- Валідно → збирає плаский список компонентів (part role, SKU, назва) → `{ valid: true, bom: [...] }`.

**`findMatchingCatalogEntry(selection, catalogEntries): CatalogEntry | null`**
- Порівнює обрану комбінацію (тип виробу, id кольору/матеріалу фасаду, id розмірів) з наявними `CatalogEntry` **за id довідників**, не за рядковими назвами — усуває проблему "кашемір/кишемір" на рівні порівняння даних.
- Збіг знайдено → BOM позначається `isStandard: true` з посиланням на існуючий SKU партії.
- Збіг не знайдено → `isStandard: false`, "нова індивідуальна комбінація"; система не блокує формування BOM, лише інформує користувача.

## UI-флоу та ролі

**Вхід:** проста hardcoded логін-форма з двома захардкодженими користувачами (manager / admin) — заглушка, без повноцінної системи авторизації.

**Роль "Менеджер" (продажі/збут):**
1. Обирає тип виробу (тумбочка) → форма конфігуратора: верх (розмір), низ (розмір), фасад (колір + матеріал з довідника), корпус (матеріал), фурнітура (multi-select зі списку).
2. Тисне "Сформувати BOM" → помилки валідації розмірів показуються інлайн біля відповідного поля; BOM не формується, поки помилки не усунено.
3. Валідний результат показує: прапорець "стандартна партія" / "індивідуальне замовлення" (залежно від збігу з каталогом), сформований BOM-список компонентів з SKU, і кнопку "Зберегти як нову каталожну позицію" (доступна лише якщо комбінації ще немає в каталозі — розширення каталогу відбувається свідомо, не хаотично).

**Роль "Адмін":**
- CRUD-екрани для довідників: кольори, матеріали, стандартні розміри, фурнітура, `PartSpec`-правила (діапазони для індивідуальних розмірів).
- Список `CatalogEntry` (стандартні партії) з можливістю перегляду/деактивації застарілих позицій.

## Верифікація

- **Юніт-тести** на доменну логіку (`validateSize`, `buildBOM`, `findMatchingCatalogEntry`): межові випадки розміру (на межі діапазону, поза межею, точний стандартний розмір), збіг/незбіг комбінації з каталогом, коректність згенерованого списку SKU.
- **Інтеграційний smoke-тест**: повний флоу через API route — обрати компоненти → отримати валідний BOM.
- **Ручна перевірка UI** (менеджер конфігурує тумбочку; адмін додає новий колір і бачить його одразу в конфігураторі) — фіксується у відео-демо.

## Agentic Engineering практики

- **AGENTS.md** — правила проєкту (стек, конвенції, розташування доменної логіки, як запускати тести) для контекст-інженерії агента.
- **SDD (Spec-Driven Development)** — цей дизайн-документ → детальний план реалізації → покрокова реалізація.
- **Верифікація замість "здається працює"** — юніт-тести доменної логіки як evals, а не лише візуальний огляд UI.
- **Maker ≠ checker** — окремий прохід `/code-review` (або окремий агент) на згенерований код перед комітом.
- **Loop engineering** — ітеративна реалізація частинами (модель даних → доменна логіка → UI) з перевіркою на кожному кроці, замість одного величезного промпту на весь проєкт.
